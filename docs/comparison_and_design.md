# セキュリティ技術の比較とシステム設計解説 (Azure & GCP)

本ドキュメントでは、マルチクラウド（Azure & GCP）環境におけるセキュアなAI API利用を実現するための3つの主要アプローチを比較し、本プロジェクトが採用した設計思想について解説します。

---

## 1. セキュリティアプローチの比較

マルチクラウド環境でAIサービス（Azure OpenAI, GCP Vertex AI等）を呼び出す際、以下の3つのセキュリティ境界モデルが考えられます。

| 評価軸 | ① 閉域網接続 (VPN / 専用線) | ② API Gateway 経由 (WAF/認証) | ③ ID連携 (キーレス OIDC) |
| :--- | :--- | :--- | :--- |
| **セキュリティ・レベル** | 🔴 **極めて高い** (インターネットに露出しない) | 🟡 **高い** (WAF・認証・IP制限でエッジ防御) | 🟢 **非常に高い** (鍵の漏洩リスクが皆無) |
| **初期構築コスト** | 🟡 中程度 (VPN等のルーティング設計が必要) | 🟢 低い (マネージドAPI Gatewayで導入可能) | 🟡 中程度 (信頼関係の設定が必要) |
| **ランニングコスト** | 🔴 **高い** (VPN/専用線ゲートウェイの時間課金) | 🟢 低い (トラフィックベースの従量課金) | 🟢 **ほぼ無料** (IDプロバイダ間の信頼関係のみ) |
| **運用複雑性** | 🔴 高い (ルーティングやDNS解決の設計が必要) | 🟡 中程度 (APIポリシーの維持が必要) | 🟢 低い (静的キーのローテーションが不要) |

### 各パターンの特徴とアプローチ

#### ① 閉域網接続 (VPN/専用線 + プライベートエンドポイント)
- **概要**: GCP VPC と Azure VNet 間を IPsec VPN トンネルまたは専用線で相互接続し、各AIサービス（Azure OpenAI等）へのアクセスを `Private Endpoint`（Azure）や `Private Service Connect`（GCP）に限定する。
- **メリット**: 攻撃者が突くインターネット経路の公開IPが存在しないため、物理的・ネットワーク的に最も強固。
- **課題**: Azure/GCPともにVPNゲートウェイは起動している時間に対して固定課金（時間あたり約$0.04〜$0.05）が発生するため、個人検証やスモールスタートではコストが膨らみやすい。

#### ② API Gateway によるセキュリティ制御
- **概要**: Azure API Management や GCP API Gateway などをパブリックに公開し、WAF（Web Application Firewall）、IPアドレス制限、OAuth2認証、レート制限（流量制御）を適用した中継APIを作成する。
- **メリット**: VPNを敷設するより遙かに低コストで導入でき、Web技術の標準的なセキュリティルールを適用できる。
- **課題**: パブリックIPのエンドポイントが存在するため、DDoS攻撃や脆弱性を突いた攻撃への対策をエッジ層で常に行う必要がある。

#### ③ ID連携によるキーレス認証 (Workload Identity)
- **概要**: GCPとAzureの間で OIDC (OpenID Connect) トークンによる信頼関係を結び、GCPで動くプログラムが「静的なAPIキー」を持たずに、Azure Entra IDにアクセスして一時的なトークンを取得し、Azure OpenAIを呼び出す。
- **メリット**: 最も脆弱になりがちな「APIキーの紛失・ハードコード」を完全に防止できる。
- **課題**: ネットワーク経路自体はパブリックであるため、①のような「インターネットを通過させない」要件には単体では対応できない。

## 2. 本プロジェクトで採用したハイブリッド設計

本プロジェクトでは、履歴書およびポートフォリオとしての技術アピール度を最大化しつつ、コストを抑えて検証できるように、以下の **ハイブリッド設計** を採用・実証しました。

```
[ ユーザープロンプト ]
       │
       ▼ 【DLP モジュール (Microsoft Presidio) によるローカル検知】
         - 個人情報 (電話番号/メール/郵便番号) の検出
         - クラウドAPIキー / シークレットの検出
         - マスキングの実行 (例: 090-XXXX-XXXX -> [PHONE_NUMBER])
       │
       ▼ 【セキュア通信経路】 (Terraformによる設計・デプロイコードの成果物)
   [GCP VPC]  ========= (IPsec VPN / 閉域通信) =========  [Azure VNet]
       │                                                      │
   [Cloud Run]  --------------------------------------->  [Private Endpoint]
 (Workload Identity)                                     (Azure OpenAI Service)
```

### 設計の3大特徴

1. **DLP（Data Loss Prevention）の統合**:
   AIへのデータ送信前に、アプリケーションローカル（Microsoft Presidio）で個人情報や認証情報をリアルタイムに検知・マスキングするモジュールを組み込みました。これにより、機密データがクラウドの外部に流出することを水際で防ぐガバナンス体制をアプリ側で保証します。
   
2. **IaC (Terraform) による閉域ネットワークのコード化**:
   Azure VNet と GCP VPC 間を VPN 接続し、Azure OpenAI に対して Private Endpoint を通じてプライベートIPアドレスでアクセスする構成をすべて Terraform でコード化しました。これにより、「設計図」としての正確性と「再現性」を持たせ、検証時のみ起動して即破棄する運用の自動化を実現しました。
   
3. **Workload Identity によるキーレス統合**:
   静的なAPIキーや証明書ファイルを使用せず、GCP のサービスアカウントと Azure Entra ID アプリケーション間を OIDC 信頼関係で結びました。これにより、APIキーのローテーション運用や環境変数漏洩のリスクを極限まで低減しています。
