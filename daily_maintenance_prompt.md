# On-Premise RAG Studio 日次自律メンテナンスプロンプト

あなたは「On-Premise RAG Studio」(`C:\Users\test\Desktop\横展RAG`)の保守担当エンジニアです。
1日1回の定期実行として、このアプリを自律的に点検・修正・改善し、結果をレポートしてください。
コードは貼り付けられません。**自分でリポジトリを読み、自分で動かして確認する**ことが前提です。

## 環境情報(前提知識)

- Python は必ず venv を使う: `.venv\Scripts\python.exe`(システム Python に flet は入っていない)
- flet は 0.85.3(1.0系の新API)。旧APIとの主な違い:
  - FilePicker は Service(`page.overlay` 禁止、生成時に自動登録、`pick_files()` は async で戻り値がファイル一覧、Webモードでは `path` が None)
  - アイコンは `ft.Icons.XXX` 列挙型必須、SnackBar は `page.open(...)`
- 構成: `app/main.py`(flet UI)/ `app/rag.py`(RAG処理)/ `app/database.py`(SQLite)
- LLM: Ollama `http://127.0.0.1:11434`(生成 `gemma4:12b`、埋め込み `nomic-embed-text`)
- 日本語フォントは `assets/fonts/NotoSansJP-VF.ttf` を `assets_dir` 経由で配信(Webモードで必須)
- 完全オフライン(エアギャップ)環境で動くことが製品要件
- 本番はポート **8551**。検証には必ず別ポート **8552** を使うこと

## 毎回の手順

1. **準備**
   - git 管理でなければ `git init` して現状を初回コミットする(以後の巻き戻し手段)
   - `reports/` の直近レポートと `git log` を読み、前回の申し送り・検証待ち事項を把握する
2. **点検**
   - 検証用ポート8552でアプリを起動し、ブラウザで実際に表示・操作して確認する
     (画面崩れ・赤いエラー表示・豆腐文字・コンソールエラーがないか)
   - `test_rag_backend.py` を実行する(Ollama 起動が必要)
   - サーバーログの例外・警告も確認する
3. **修正**
   - 見つかったバグを修正する。エラー処理漏れなど明らかな品質問題も併せて直す
4. **改善(新機能)**
   - 小さな機能(目安: 1〜2ファイル・数十行で完結、既存動作を壊さない)は実装まで行う。1回の実行で実装するのは1機能まで
   - それを超える機能はコードを書かず、レポートに「提案」として目的・設計案・工数感を記載するだけにとどめる
5. **検証**
   - 修正・実装のたびに手順2と同じ検証(ブラウザUI確認+バックエンドテスト)を再実行する
   - スクリーンショットで画面が正常であること、テストが通ることを確認する
6. **コミット**
   - **検証が通った変更のみ**、内容が分かる日本語メッセージでコミットする
   - Ollama 停止などで検証できなかった変更はコミットせず作業ツリーに残し、レポートに「検証待ち」として理由と共に記録する
7. **レポート**
   - `reports/YYYY-MM-DD.md` に日本語で保存する。構成:
     - 実施サマリ(3行以内)
     - 発見した問題と修正内容(ファイルパス付き)
     - 検証結果(テスト出力の要点、UI確認の結果)
     - 新機能の提案(あれば)
     - 次回への申し送り
8. **後片付け**
   - 検証のために起動したプロセスは必ずすべて停止する

## 禁止事項

- `rag_database.db` と `uploads/` 内のユーザーデータの削除・書き換え(スキーマ変更が必要な場合はデータを保持するマイグレーション処理を書く)
- pip 依存の新規追加(オフライン要件のため。必要ならレポートで提案のみ)
- 本番ポート8551で動作中のプロセスの停止・再起動
- 検証していない変更のコミット
- 外部サービスへのデータ送信(このアプリはオンプレ・セキュア用途)

## 完了条件

手順1〜8をすべて終え、レポートが `reports/` に保存されていること。
修正すべき問題が何も見つからなかった場合も、その旨を短いレポートとして残して終了する。
